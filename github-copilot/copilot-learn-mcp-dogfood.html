The loop

Five steps from cron tick to drafted PR

Everything runs inside the repository. No external SaaS dashboard, no out-of-band workflow. The agent's identity, scope, and outputs are all inspectable in git history.

1

Scheduled trigger

Cron in accuracy-review.yml fires weekly (Thursday 07:00 UTC), caps at 10 issues per run.

2

Issues created

One issue per stale page, labeled copilot-review, assigned to Copilot via a user-scoped PAT.

3

Copilot picks up

Cloud agent detects the assignment, spins an ephemeral Actions environment, reads the page.

4

Verify against Learn

Checks current product state on Microsoft Learn (via MCP when configured, or web fetch) and compares to the page's claims.

5

Draft PR

If stale claims found, opens a copilot/accuracy-review-* branch with edits + rationale. No auto-merge.

Why this shape works

  • Zero human effort per review cycle. The cron, the assignment, and the PR draft all happen without anyone on keyboard.
  • Microsoft Learn is a first-party source. Not general web search — a Microsoft-owned documentation surface that tracks renames, GA dates, deprecations.
  • Evidence stays in the repo. Every review run leaves an issue, a branch, and (if stale) a PR. The audit trail is git log.
  • Humans still approve. Nothing auto-merges. Every PR needs a reviewer to accept the fact-check.

What it doesn't do

  • Not free. Two workflows totalling ~700 lines of YAML + a PAT secret + Copilot seats + Actions minutes. Worth it here because it amortizes across every future page.
  • Not perfect. Copilot can miss subtle inaccuracies or flag false positives. Review quality still depends on the reviewer.
  • Not portable as-is. This pattern fits Microsoft-employee content about Microsoft products. Your mileage will vary for non-MS domains.
  • Not a substitute for SME review. A correctly-cited stale fact is still stale — Learn itself lags the live product in edge cases.
