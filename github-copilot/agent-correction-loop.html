When the agent's PR is wrong

Every async coding agent ships broken first drafts sometimes. The question is how you tell it to try again — and whether that conversation survives in the repo or lives somewhere else.

The loop

Four tools, four correction shapes

What a reviewer has to do to get a v2. The cheaper this loop is, the more viable the "assign it to the bot" pattern becomes for real work.

GitHub Copilot cloud agent

Native PR surface
1
Comment in the PR. @copilot address the null-check on line 42.
2
Agent wakes on the same branch. New commit pushed, same PR.
3
Review comments resolve inline. Standard GitHub review UI.
4
Conversation stays in the PR. Every iteration is git log-visible.

Claude Code Action

@claude-triggered workflow
1
Comment with @claude. Workflow fires on the event.
2
Action runs on your runner. Progress tracked in the comment itself (checkboxes that update live).
3
New commit on the PR branch. Same PR, one Actions run per @claude trigger.
!
Each @claude = Actions minutes + Anthropic API tokens. Chatty reviewers cost more.

OpenAI Codex in GitHub

@codex PR comment
1
Comment @codex <task>. Codex reacts with 👀.
2
Codex opens a cloud task. Uses the PR as context.
3
Updates posted back to the PR. Review step is @codex review, a separate command.
!
Full task state lives in ChatGPT Codex UI. PR sees the outputs; the deliberation is elsewhere.

Cursor Cloud Agents

@cursor PR comment (or Cursor web)
1
Comment @cursor on the PR — or kick off a new run from Cursor web, Slack, Linear, or API.
2
Agent clones, modifies, pushes. Cloud VM with own tools, MCP, hooks.
3
Optional: follow up from the IDE. Cursor Desktop can view and continue cloud-agent runs.
!
Run history in Cursor dashboard. PR shows outputs; reasoning lives in Cursor's UI.
Practical pattern

What a good correction comment looks like

All four respond better to concrete, diff-specific feedback than to "this is wrong, try again." Pattern below works on any of them.

Weak
@copilot this doesn't work, please fix it
Better
@copilot the rate-limit check in auth.ts:42 uses user.id as the key but that's null for unauthenticated requests. Use request.ip as the fallback and add a test for the anonymous case.

What makes a good correction loop

  • One surface. Reviewers shouldn't tab between GitHub and a vendor dashboard to give feedback.
  • Cheap iteration. The cost of a "try again" comment should not change reviewer behavior.
  • State in the PR. Future maintainers reading git log + PR comments should see the whole conversation.
  • Stop after 2-3 rounds. If it's still wrong, take over manually; iteration cost exceeds writing the fix.

What this piece doesn't compare

  • Not iteration quality. No claim about which tool's v2 is better than its v1.
  • Not response latency. All four run async; seconds-to-response varies by load, not product.
  • Not custom prompts. Each tool has fine-grained prompt-tuning features; this piece only covers the default review-driven loop.
  • Not trigger-phrase overrides. Claude lets you rename @claude; Codex and Cursor likewise customize.
